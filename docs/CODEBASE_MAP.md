### Execution Runners

**Spawn Runner** (`src/spawn-runner.ts`) ‚Äî default for exec mode, provider-routed
| File | Purpose | Tokens |
|------|---------|--------|
| `src/spawn-runner.ts` | Detached child_process runner with provider-specific launchers | 1,376 |

- Routes to `buildSpawnLauncher` (OpenAI) or `buildGeminiLauncher` (Gemini) based on `options.provider`
- Both launchers use `PIPESTATUS[0]` (not `$?`) to capture provider exit code, not `tee` exit
- OpenAI launcher: `codex exec -p '' --model <model> -r <reasoning> -s <sandbox> -c model_reasoning_summary=concise`
- Gemini launcher: `gemini -p '' <sandboxFlag> -m <model> -o text`; sandbox mapping: `read-only` ‚Üí `-s`, others ‚Üí `--yolo`
- Generates `.sh` launcher per job; spawns `bash <launcher>` detached with `unref()`
- `set -uo pipefail` ensures exit code write even if `tee` fails
- `isProcessAlive`: uses `process.kill(pid, 0)` probe
- Launcher files written with `0o700` permissions

## Conventions

- **Atomic writes**: All JSON and signal file writes use `atomicWriteFileSync` (tmp + `renameSync`)
- **Argv-safe tmux**: All `spawnSync` calls use argv arrays ‚Äî no shell string interpolation
- **Shell quoting in launchers**: `shellQuote()` uses `'\''` technique for bash-safe embedding
- **Bun-only**: `bun:sqlite` used throughout; not portable to Node.js
- **ES modules**: `"type": "module"` in `package.json`; dynamic `require()` used in store factory to avoid circular imports
- **Permissions**: Jobs dir created with `0o700`; job JSON files written with `0o600`

## Gotchas

- **`watcher.ts:updateJobTurn` bypasses store abstraction** ‚Äî writes `.json` directly, creating potential dual-write inconsistency with SQLite
- **`session-parser.ts` duplicates ANSI stripping** ‚Äî `extractSessionId` has its own regex instead of reusing `output-cleaner.ts`
- **`findSessionFile` matches by substring** ‚Äî potential false-positives for similar session IDs
- **Idle detection regex** (`? for shortcuts`) can briefly appear during Codex TUI redraws; 30s grace period mitigates false-positives
- **`store/_store` singleton never resets** ‚Äî changing `CODEX_AGENT_STORAGE` env after first `getStore()` call has no effect in same process
- **`codex-agent` wrapper defaults to `~/.codex-orchestrator/bin/`** (upstream install), not local dev path ‚Äî use `bun run src/cli.ts` for local testing
- **`agents.log.template` is deprecated** ‚Äî SKILL.md explicitly discourages shared agents.log to avoid multi-Claude race conditions; use `jobs --json` instead
- **50MB buffer limit** for `captureFullHistory`
- **hardcoded `sleep 1`** in `tmux.ts` before sending prompt ‚Äî fragile TUI sync, necessary for tmux/Codex startup timing
- **`DualStore.list()` returns SQLite only** ‚Äî JSON-only jobs invisible until backfill completes
- **Exit code behavior change** ‚Äî non-zero exit + completion marker now correctly becomes `failed` (previously was `completed`); existing workflows relying on marker override will see different behavior
- **Gemini enrichment is always `null`** ‚Äî session-parser only handles Codex JSONL; `getJobsJson` skips enrichment for non-openai providers
- **Gemini silent hang risk** ‚Äî ambiguous prompts via stdin may cause Gemini to wait indefinitely; mitigated by `geminiHardMaxRuntimeMinutes` hard kill (default 30 min)
- **`provider` field backward compat** ‚Äî old jobs without `provider` field use `?? "openai"` fallback everywhere; SQLite v3 migration auto-defaults existing rows to `'openai'`
- **CHROME_CUT_MARKERS regex `lastIndex`** ‚Äî explicitly reset inside `truncateAtFirstMarker` (needed for non-global regexes with `exec`)

## Navigation Guide

**To add a new CLI command**: Edit `src/cli.ts` `switch(command)` block + add relevant logic in `src/jobs.ts`

**To add a new job field**: Update `Job` interface in `src/jobs.ts` ‚Üí update `SqliteStore` schema + migration in `src/store/sqlite-store.ts` (increment `SCHEMA_VERSION`) ‚Üí update `JsonStore` if needed

**To change Codex prompt behavior**: Edit `src/prompt-constraints.ts` (XML constraint blocks) or `plugins/codex-orchestrator/skills/codex-orchestrator/SKILL.md`

**To add a new storage backend**: Implement `JobStore` interface from `src/store/job-store.ts` ‚Üí register in `src/store/index.ts` factory

**To change exec runner**: Set `CODEX_AGENT_EXEC_RUNNER=tmux` for legacy tmux exec mode; or modify `src/spawn-runner.ts`

**To add a new provider**: Add to `Provider` type in `src/config.ts` ‚Üí add launcher in `src/spawn-runner.ts` ‚Üí add fail-fast validation in `src/jobs.ts:startJob` ‚Üí add cleaner in `src/output-cleaner.ts` ‚Üí bump SQLite schema if needed ‚Üí add patterns in SKILL.md. Currently uses `if/else`; refactor to `ProviderAdapter` interface when 3rd provider added.

**To debug job completion**: Check `~/.codex-agent/jobs/<id>.exitcode` (spawn) or tmux pane for `[codex-agent: Session complete` marker (tmux exec). Exit code is authoritative: non-zero = failed regardless of completion marker.

**To run multi-provider analysis**: Use Pattern A/B/C from SKILL.md. Start two jobs with `--provider openai` and `--provider gemini`, await both, then synthesize with anonymous Alpha/Beta relabeling per Synthesis Protocol.

**To update the plugin skill**: Sync changes to 3 locations ‚Äî source (`plugins/codex-orchestrator/skills/.../SKILL.md`), marketplace clone (`~/.claude/plugins/marketplaces/codex-orchestrator/`), plugin cache (`~/.claude/plugins/cache/codex-orchestrator/`)
